# 分析 Analysis

## 1. 使用者與情境

| 使用者類型 | 情境 | 關注點 |
|---|---|---|
| 新玩家 | 透過分享 URL 進入，第一次玩 | 第一印象、上手速度、不需註冊 |
| 回流玩家 | 數天後回來繼續玩 | 存檔仍在、進度沒丟、能從上次斷點接續 |
| 短時間玩家 | 通勤 / 工作空檔玩 10–20 分鐘 | 可隨時中斷、自動存檔、不需 commit 大段時間 |
| 長時間玩家 | 假日連玩數小時 | 內容耐玩、不會卡關、有目標感 |
| 行動裝置玩家 | 手機 / 平板瀏覽器 | 觸控操作、螢幕適配、流量友好 |
| 桌面玩家 | 鍵盤 + 滑鼠 | 鍵盤快捷鍵、清晰 UI |

## 2. 核心行為流

1. **進入** → 標題畫面 → 新遊戲 / 繼續 / 設定
2. **新遊戲** → 主角命名 → 開場劇情 → 起始地圖
3. **探索** → 方向鍵 / 觸控搖桿移動 → 觸發事件（遇敵、NPC、寶箱、場景切換）
4. **戰鬥** → 進場動畫 → 行動選單（攻擊 / 魔法 / 道具 / 防禦 / 逃跑）→ 結算 → 經驗 / 金幣 / 道具
5. **成長** → 升等 → 數值提升 → 解鎖技能
6. **村莊機制** → 商店買賣、旅館回血、NPC 對話、接任務
7. **存檔** → 手動存檔點 + 自動存檔（每場戰鬥後、每張地圖切換時）

## 3. 邊界案例與風險

- **存檔遺失**：清除瀏覽器資料 / 換瀏覽器 / 換裝置 → 純 LocalStorage 無法跨裝置同步。需思考是否提供「匯出存檔 JSON / 匯入存檔」備援
- **多分頁同步**：玩家開兩個分頁玩同一個遊戲 → 存檔互相覆蓋。需 lock 或警告
- **意外關閉**：玩家在戰鬥中關掉瀏覽器 → 是否回滾到戰鬥前？
- **資源載入**：立繪 / 音樂太大 → 首次進入要等很久。需 lazy loading / 漸進載入
- **行動裝置適配**：解析度差異大、觸控操作 vs 鍵盤
- **遊戲平衡**：怪物過強卡關 / 過弱無趣 → 需要可調的數值表
- **作弊**：純前端任何數值都能被改 → 接受作弊，不做反作弊（單人遊戲，自虐 OK）
- **無網路使用**：理論上純前端應可離線玩 → 是否做 PWA / Service Worker

## 4. 約束條件

- **時間**：作者業餘時間，無明確 deadline。建議 MVP scope 控制在 1–2 個月可玩出原型
- **技術**：純前端，無後端伺服器 → 無雲端存檔、無排行榜、無多人
- **儲存**：LocalStorage 5–10 MB / IndexedDB 較大 → 存檔資料量需精簡
- **美術成本**：2D 卡通立繪需要 AI 生成（如 Stable Diffusion / Midjourney）或購買素材，無原畫師
- **音樂 / 音效**：使用 CC0 / 公開素材
- **瀏覽器支援**：現代瀏覽器（最近 2 年版本的 Chrome / Firefox / Safari / Edge），不支援 IE / 舊版

## 5. MVP vs 擴充

- **MVP（先做出可玩原型）**：1 個職業、3–5 種怪、1 個小村莊 + 1 個小迷宮、戰鬥系統、存讀檔、基本劇情
- **擴充項（後續加）**：多職業、隊伍、地圖擴展、支線任務、成就、PWA 離線、雲端存檔（若加後端）
